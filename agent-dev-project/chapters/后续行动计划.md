### 2.1 评估脚本

面试官必问："你的 RAG 系统效果怎么样？"

不能回答"肉眼看还行"。需要量化指标。

**新建文件**：`eval_retrieval.py`

```python
"""
检索效果评估

三个核心指标：
  Hit Rate@K   = 正确答案出现在 Top-K 中的查询数 / 总查询数
  MRR          = 平均 (1 / 正确答案第一次出现的排名)
  Recall@K     = Top-K 中相关文档数 / 总相关文档数
"""

# 测试集：20 条 (query, 正确答案关键词列表)
test_cases = [
    {"query": "小米SU7的续航是多少", "answer_keywords": ["小米", "SU7", "续航"]},
    {"query": "25万以内的大空间SUV", "answer_keywords": ["理想L6", "海狮", "极氪"]},
    {"query": "便宜的城市通勤代步电车", "answer_keywords": ["海豚", "海鸥", "宏光MINI"]},
    # ... 共 20 条
]

# 对每条 query 跑三路检索，算指标
def evaluate(retriever, test_cases, top_k=3):
    hit = 0
    mrr_total = 0.0
    for case in test_cases:
        results = retriever.search(case["query"], top_k)
        # 检查 Top-K 里是否命中 answer_keywords
        for rank, doc in enumerate(results, start=1):
            if any(kw in doc["content"] for kw in case["answer_keywords"]):
                hit += 1
                mrr_total += 1.0 / rank
                break
    return {
        "hit_rate": hit / len(test_cases),
        "mrr": mrr_total / len(test_cases),
    }

# 输出三路对比表
# 关键词检索 Hit Rate: 68%,  MRR: 0.45
# 向量检索   Hit Rate: 82%,  MRR: 0.58
# 混合 RRF   Hit Rate: 89%,  MRR: 0.66
```

**实施步骤**：
1. 手写 20 条 (query, 正确答案关键词) 测试对
2. 导入 `retrieval_test.py` 中的三类检索器
3. 跑评估，打印对比表
4. 分析：哪类查询混合检索提升最大？哪类仍然是瓶颈？

### 2.3 Streamlit 部署

面试官必问："你的系统我能试一下吗？"

```python
import streamlit as st
from chapters.retrieval_test import v_idx, bm25_idx, hybrid_rrf
from chapters.naive_rag import build_prompt, ask_llm

st.set_page_config(page_title="新能源汽车 RAG 助手", page_icon="🚗")
st.title("新能源汽车知识库问答")

query = st.text_input("请输入你的问题", placeholder="例如：25万以内的大空间SUV推荐")

if query:
    with st.spinner("检索中..."):
        # 混合检索
        vec_candidates = v_idx.search(query, top_k=6)
        bm25_candidates = bm25_idx.search(query, top_k=6)
        results = hybrid_rrf(vec_candidates, bm25_candidates)

    st.subheader("检索到的参考资料")
    for i, (score, doc) in enumerate(results):
        with st.expander(
            f"{'⭐' if i == 0 else '📄'} [{doc['source']}] (RRF: {score:.4f})"
        ):
            st.write(doc["content"])

    if st.button("🤖 调用 LLM 回答", type="primary"):
        context = "\n\n---\n\n".join([doc["content"] for _, doc in results])
        messages = build_prompt(query, context)
        answer = ask_llm(messages)
        st.subheader("回答")
        st.write(answer)
```

## 三、技术补缺路线图

### 优先级一：LangChain 认知（半天）

**为什么**：面试必问"你用过什么框架"。不是因为你必须用框架，是因为面试官用过。

**怎么学最快**：不要看 LangChain 文档（太乱）。拿你已经写完的代码对照理解：

| 你手写的 | LangChain 等价物 | 一句话 |
|---------|-----------------|--------|
| `load_data()` | Document Loader | 从各种数据源加载文档 |
| `chunk_sections()` | Text Splitter | 把长文档切成小块 |
| `embed_documents()` | Embeddings | 文本转向量 |
| `VectorIndex` | VectorStore | 向量索引 + 相似度搜索 |
| `search()` → `build_prompt()` → `ask_llm()` | RetrievalQA / Chain | 检索 → 拼 prompt → 调 LLM |
| `hybrid_rrf()` | EnsembleRetriever | 多路检索结果融合 |

**面试时这句话**："我没用 LangChain，但我理解它的每一层在做什么，而且我手写过底层。LangChain 的 VectorStore 对应我写的 VectorIndex，EnsembleRetriever 对应我的 RRF 融合。"

比你说"我用 LangChain 做了个项目"强十倍。

### 优先级二：Reranker 重排序（2h，直接加到你现有项目）

**解决了什么**：检索拉回来的 Top-K，向量分高 ≠ 真的相关。Reranker 用 CrossEncoder 做精排，把候选文档逐个和 query 算真正的相关性。

**如何加到现有项目**：

```python
# 1. 安装
# pip install sentence-transformers

# 2. 加载 Reranker 模型（和 BGE 同一个作者，可直接用）
from sentence_transformers import CrossEncoder
reranker = CrossEncoder("BAAI/bge-reranker-base", max_length=512)

# 3. 在混合检索之后加一步重排
def rerank(query, candidates, top_k=3):
    """对候选文档用 CrossEncoder 精排"""
    pairs = [[query, doc["content"]] for doc in candidates]
    scores = reranker.predict(pairs, show_progress_bar=False)
    # 或者用模型自带的 compute_score
    ranked = sorted(zip(scores, candidates), key=lambda x: x[0], reverse=True)
    return ranked[:top_k]

# 4. 管线变为：粗排(混合RRF top_k*2) → 精排(Reranker) → LLM
```

**效果**：粗排保证召回率（不错过相关文档），精排保证准确率（LLM 收到最相关的上下文）。

### 优先级三：Query 改写（2h）

**解决了什么**：用户问"上次那个车怎么样"——没有上下文。用 LLM 把省略/模糊的 query 扩展成完整的检索 query。

```python
QUERY_REWRITE_PROMPT = """结合对话历史，把用户问题改写成独立的、信息完整的检索查询。

对话历史：
{chat_history}

用户问题：{query}

只输出改写后的查询，不要解释。"""

def rewrite_query(query, chat_history):
    """把依赖上下文的问题改写成独立查询"""
    messages = [
        {"role": "user", "content": QUERY_REWRITE_PROMPT.format(
            chat_history=chat_history,
            query=query
        )}
    ]
    return ask_llm(messages)

# 例子：
# chat_history: "推荐一款大空间家用SUV" → "理想L6，24.98万起"
# query: "那它的续航呢"
# 改写后: "理想L6的续航里程"
```

### 优先级四：Chunk 父子结构（1h）

**解决了什么**：小块检索精度高但丢失上下文，大块上下文全但检索精度低。

**方案**：检索用小 chunk，返回时带上它所属的大 chunk（父块）。

```python
# 建库时：为每个 chunk 指定 parent_id
documents.append({
    "content": chunk["content"],     # 小块，用于检索匹配
    "parent_content": full_section,  # 大块，用于喂给 LLM
    "chunk_id": chunk["chunk_id"],
    "parent_id": section["section_id"],
    ...
})

# 检索时：用 content 匹配，用 parent_content 喂 LLM
```

## 四、学习方法：如何从"懂原理"到"能写代码"

### 核心问题诊断

```
你已知的：                   你卡住的：
"这里要做向量检索"          → faiss？numpy？怎么调？
"这里要打分归一化"          → Min-Max 归一化公式是什么？
"BM25 有 IDF 和词频饱和"    → log 底数是多少？参数 k1/b 默认值？
"融合用 RRF 不用加权"       → RRF 的 k 是什么？rank 从 0 还是 1 开始？
```

**问题出在"概念 → API"的翻译环节，不是理解环节。**

### 解法一：3-Round 还原练习

对着你自己的 markdown 文档，用三个 Round 练习把"逻辑描述"还原成"可运行代码"：

| | Round 1 | Round 2 | Round 3 |
|---|---|---|---|
| **看的材料** | 你的 markdown 文档（含代码骨架） | 只有架构图/数据流图 | 什么都不看 |
| **卡住了** | 翻项目源代码 | 查 API 文档 / Google | 硬写，记错就记错 |
| **目的** | 逻辑 → 代码翻译训练 | 模拟真实独立开发 | 暴露盲区 |
| **时间** | 1~2h | 1~2h | 30min |
| **什么时候算过** | 不看源码能跑通 | 只靠 API 文档能跑通 | diff 对比无遗漏暗坑 |

Round 2 看的材料：只有架构图中一句话——「第四章 BM25 类：DF 统计、IDF、饱和词频、长度归一化」

Round 3：开空白 `.py` 文件，凭记忆写完整 BM25 类。写完后和源码 diff，暴露出来的遗漏点就是你的盲区，记到 snippets.md 里。
