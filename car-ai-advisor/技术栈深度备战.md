## 1. ReAct Agent 循环

### 原理图解

```
                ┌─────────────────────────────────────────┐
                │           ReAct 循环（非流式）            │
                │                                         │
  用户提问 ───► │  组装 messages ──► 调 LLM(带 tools)      │
  +历史+提示    │         │                               │
                │         ▼                               │
                │   有 tool_calls? ──是──► 执行工具        │
                │         │                │              │
                │         否               ▼              │
                │         │          结果回填 messages ──┐ │
                │         ▼                              │ │
                │   得到最终回答 ◄───────────────────────┘ │
                └─────────────────────────────────────────┘
                          │
                          ▼
              字符级流式 yield 给前端（SSE）
```

三个护栏：**最大轮数 6 轮**（防失控）、**时间预算 95 秒**（防慢 API 拖死前端）、**无工具收尾调用**（预算耗尽时不传 tools 强制模型输出文本，保证用户永远看到回答）。

### 核心代码框架（简化骨架，能默写到这个程度）

```python
MAX_ITERATIONS = 6
LLM_BUDGET_SECONDS = 95

async def stream_chat(self, query, history):
    messages = [{"role": "system", "content": SYSTEM_PROMPT},
                *history,
                {"role": "user", "content": effective_query}]
    final_content = ""
    t0 = time.monotonic()

    for iteration in range(MAX_ITERATIONS):
        if time.monotonic() - t0 > LLM_BUDGET_SECONDS:
            break                                  # 时间预算耗尽 → 收尾
        response = await self.llm.chat.completions.create(
            model=self.model, messages=messages, tools=TOOL_SCHEMAS,
            tool_choice="required" if iteration == 0 else "auto",
        )
        msg = response.choices[0].message
        tool_calls = parse_native(msg) or parse_xml_fallback(msg.content)

        if tool_calls:                             # 拦截 → 执行 → 回填 → 继续
            for tc in tool_calls:
                result = await self.executor.execute(tc["name"], tc["arguments"])
                messages.append({"role": "tool", "tool_call_id": tc["id"],
                                 "content": result})
            continue

        final_content = strip_all_xml(msg.content)  # 无工具调用 → 最终回答
        break

    if not final_content:                          # 护栏：无工具收尾调用
        wrap = await self.llm.chat.completions.create(
            model=self.model, messages=messages)    # 不传 tools，只能输出文本
        final_content = strip_all_xml(wrap.choices[0].message.content)

    for ch in final_content:                       # 字符级流式输出
        yield {"type": "token", "content": ch}
```

### 必背追问

- **循环什么时候退出？** 三种：模型不再发起工具调用（正常）、达到 6 轮上限、95 秒时间预算耗尽
- **为什么首轮 required 后续轮 auto？** 首轮强制先查知识库防编造；工具结果回填后必须允许模型说"够了，直接回答"
- **工具执行失败怎么办？** 异常捕获后返回纯文本错误消息回填，让模型基于"工具失败"继续推理，而不是整个对话崩掉

## 2. Function Calling（工具调用机制）

### 原理图解

```
开发者                        LLM                        你的代码
  │                            │                            │
  │── tools: JSON Schema ─────►│                            │
  │   (函数名+参数+描述)        │                            │
  │                            │── 输出调用意图(文本) ──────►│
  │                            │   {"name":"compare_cars",  │ 解析+校验参数
  │                            │    "arguments":{...}}      │
  │                            │                            │── 真正执行函数
  │                            │◄── role:"tool" 回填结果 ───│
  │                            │── 基于结果生成最终回答 ────►│── 给用户
```

**关键认知：模型从不执行任何代码**，它只生成"我想调用某个函数"的结构化文本，执行、权限、参数校验全部在客户端工程层。

### 核心代码框架

```python
TOOL_SCHEMAS = [{                      # ① 用 JSON Schema 描述工具
    "type": "function",
    "function": {
        "name": "compare_cars",
        "description": "对比两款车型的参数、价格、续航",
        "parameters": {
            "type": "object",
            "properties": {
                "car_a": {"type": "string", "description": "车型A规范名"},
                "car_b": {"type": "string", "description": "车型B规范名"},
            },
            "required": ["car_a", "car_b"],
        },
    },
}]

# ② 模型返回调用意图 → ③ 客户端执行 → ④ 以 role="tool" 回填
if msg.tool_calls:
    for tc in msg.tool_calls:
        args = json.loads(tc.function.arguments)     # 解析模型给的参数
        result = await executor.execute(tc.function.name, args)
        messages.append({"role": "assistant", "tool_calls": [...]})
        messages.append({"role": "tool",
                         "tool_call_id": tc.id,      # 用 id 对应调用与结果
                         "content": result})
```

### 必背追问

- **模型怎么"学会"调工具的？** SFT 阶段用大量工具调用格式的数据训练；推理时 schema 在 prompt 里，模型照格式生成
- **参数传错怎么办？** 三层：schema 描述写清示例 → 执行层强校验并返回可读错误（模型下一轮自我修正）→ 车名"四级解析"兜底归一化
- **并行调用？** 无依赖的调用（同时查两款车参数）可一轮发出、并发执行一起回填；有依赖的（查价格→算落地价）必须串行多轮

## 3. RAG 全链路（FAISS + BM25 + RRF + BGE 精排）

### 原理图解

```
【建库阶段（离线）】
  汽车文档 ──► 切块(chunking) ──► BGE嵌入 ──► FAISS索引(IndexFlatIP)
                                        └──► BM25倒排(jieba分词)
【查询阶段（在线）】
  "宋L续航怎么样" ──┬──► embed_query ──► FAISS向量召回 Top-N ──┐
                    └──► jieba分词 ──► BM25关键词召回 Top-N ──┤
                                                               ▼
                                              RRF融合: score=Σ 1/(60+rank)
                                                               ▼
                                              BGE Reranker 交叉精排
                                                               ▼
                                          Top-5 注入 prompt ──► LLM 生成(有出处)
```

### 核心代码框架

```python
# ① 向量召回：归一化后内积 = 余弦相似度
class VectorIndex:
    def __init__(self, documents):
        embeddings = np.stack([d.embedding for d in documents])  # 已归一化
        self.index = faiss.IndexFlatIP(dim)       # 内积索引，~100条暴力精确够用
        self.index.add(embeddings)

# ② BM25：IDF加权 + 词频饱和 + 长度归一化（k1=1.5, b=0.75）
def _score_one(self, q_tokens, doc_tokens):
    for t in q_tokens:
        f = doc_tokens.count(t)
        tf_sat = f*(self.k1+1) / (f + self.k1*(1-self.b + self.b*dl/self.avgdl))
        score += self.idf(t) * tf_sat             # idf = log(1+(N-n+.5)/(n+.5))

# ③ RRF 融合：按名次不按分数，免疫两边量纲差异
def hybrid_rrf(vec_results, bm25_results, k=60, top_k=5):
    for rank, (_, doc) in enumerate(vec_results):
        scores[doc.chunk_id] += 1.0 / (k + rank + 1)
    for rank, (_, doc) in enumerate(bm25_results):
        scores[doc.chunk_id] += 1.0 / (k + rank + 1)
    return sorted_top_k(scores)
```

### 必背追问

- **为什么混合检索？** 向量擅语义（"适合家用的车"），弱于精确型号/数字（"宋L 662km"）；BM25 正好相反——互补
- **为什么 RRF 不直接加权分数？** 余弦相似度(0~1)和 BM25 分(0~几十)量纲完全不同，加权系数没法调；RRF 只用名次，`k=60` 是论文通用值不需调参
- **规模上去后 FAISS 怎么换？** 十万级换 IndexIVFFlat（倒排聚类加速），百万级换 IndexHNSWFlat（图索引）——代码注释里就写了这个演进路径，主动讲是加分项
- **为什么还要 Reranker？** 双塔是"分别编码再比相似"，交互信息少；Reranker 交叉编码把 query+文档拼一起算相关性，精度高但慢——只排粗排 Top-N，精度与速度的平衡

## 4. SSE 流式输出 + XML 协议拦截（招牌题）

### 原理图解

```
错误架构（泄露根源）：
  LLM ──stream──► 直接透传前端     ✗ 用户看到 <tool_call>{"name":... 碎片
                                     且标记跨 chunk 截断，正则根本堵不住

本项目架构（分层拦截）：
  ┌─ ReAct 循环阶段：非流式 ─────────────────────────┐
  │  LLM 输出完整收集 → 判断有无工具调用              │
  │  有 → 执行工具（前端只看到"正在查询…"状态）        │
  │  无 → 最终回答再过一遍 6 层 XML 清理安全网         │
  └──────────────────────────────────────────────────┘
                    │  纯净文本
                    ▼
  逐字符 yield ──► SSE: data: {"type":"token","content":"推"} ──► 前端边收边渲染
```

### 核心代码框架

```python
# ① 6 层清理安全网（顺序不能乱：先大块后碎片）
def strip_all_xml(text):
    text = INVOKE_BLOCK_RE.sub("", text)      # 1.完整<invoke>块(含内容)
    text = TOOL_CALLS_RE.sub("", text)        # 2.<tool_calls>包装块
    text = TAG_RE.sub("", text)               # 3.所有独立标签(含hy-前缀/DSML/全角管道)
    text = FRAGMENT_RE.sub("", text)          # 4.未闭合片段
    text = TAIL_RE.sub("", text)              # 5.被max_tokens截断的半个尾标签
    text = BARE_DSML_RE.sub("", text)         # 6.裸DSML管道标记
    return text.strip()

# ② SSE 事件流：前端只收 token/error，工具事件被过滤
async def sse_generator(agent, query, history):
    async for event in agent.stream_chat(query, history):
        if event["type"] == SSE_TOKEN:                 # 纯文本才发
            yield f"data: {json.dumps(event)}\n\n"     # SSE 格式: data:...\n\n
```

### 必背追问（这题几乎必被深挖）

- **为什么不在前端过滤？** ① 标记跨 chunk 截断（`<tool_` + `call>`），单片正则必漏；② 协议逻辑收敛在后端，前端保持"所见即所得"，换个前端也不用重写
- **为什么工具阶段非流式？** 要判断"这段输出是工具调用还是最终回答"，必须看到完整内容才能决策——流式中途无法回头
- **代价是什么？怎么补偿？** 工具执行期间无字符输出，前端用"正在查询知识库…"状态提示；最终回答阶段恢复逐字流式，体感不差
- **实测遇到过的变体？** DeepSeek 输出过 `<｜｜DSML｜｜invoke>`（全角双管道）、`hy-invoke` 前缀、截断半标签——正则都兼容了，25 组对抗用例回归

## 5. Python 异步与流式 HTTP

### 原理图解

```
同步 vs 异步（IO 密集型场景）：
  同步: 请求1 ██████░░等待LLM░░██████
        请求2                    ██████░░等待░░██████     ← 排队干等
  异步: 请求1 ████░░等待(让出)░░░████
        请求2     ████░░等待░░████                        ← 单线程并发

await 点 = 让出控制权：等 LLM API / 等 Redis / 等文件 IO 时，
事件循环去服务其他请求，单机并发能力数倍提升。
```

### 核心代码框架

```python
from openai import AsyncOpenAI

class CarAdvisorAgent:
    def __init__(self):
        self.llm = AsyncOpenAI(api_key=..., base_url=...,
                               timeout=60.0, max_retries=1)

    async def stream_chat(self, query, history):       # 异步生成器
        response = await self.llm.chat.completions.create(...)  # 不阻塞事件循环
        ...
        for ch in final_content:
            yield {"type": "token", "content": ch}     # 逐字符产出

# FastAPI 侧：SSE 响应直接把异步生成器接出去
@router.post("/chat/stream")
async def chat_stream(req: ChatRequest):
    return StreamingResponse(sse_generator(agent, req.query, history),
                             media_type="text/event-stream")
```

### 必背追问

- **为什么异步适合这个场景？** 耗时大头是等 LLM API（IO 密集），async 等待时让出 CPU 服务其他连接；CPU 密集场景（如 Reranker 推理）异步无收益，要用多进程/线程池
- **`max_retries=1` 为什么改默认？** 默认重试 2 次，慢 API 时卡顿放大到 3×60s；限制重试保体验——讲这个说明你真的压测过
- **超时怎么设？** LLM 单次 60s + 整体预算 95s + 前端 120s，三层超时由内向外递增，每层都有兜底

## 6. FastAPI

**原理**：基于 Starlette（ASGI 异步框架）+ Pydantic（数据校验）。请求进来后 Pydantic 模型自动校验/反序列化，依赖注入系统（Depends）统一处理认证、限流等横切逻辑，自动生成 OpenAPI 文档。

**选型权衡**：
- vs Flask：Flask 是同步 WSGI，SSE 长连接会占满 worker；FastAPI 原生 async，长连接不占线程
- vs Django：Django 全家桶太重，本项目只需要 API 层，不需要 ORM/Admin/模板

**高频追问**：
- "依赖注入解决什么？" → 认证/限流写成 Depends 挂在路由上，业务函数零侵入，测试时可替换实现
- "Pydantic 干嘛的？" → 请求体自动校验类型，非法请求 422 直接拒掉，错误不进业务代码

---

## 7. Redis

**原理**：内存 KV 存储，单线程事件循环处理命令。本项目用 `SET session:{id} {json} EX 86400` 存会话历史，TTL 到期自动清。

**选型权衡**：
- vs 内存字典：进程重启即丢、多实例不共享、无过期机制——开发期可以，生产不行
- vs 数据库：会话是高频读写+天然过期的数据，落库是杀鸡用牛刀
- 无 Redis 环境用 fakeredis 兜底：**接口一致零改动**，本地开发友好

**高频追问**：
- "Redis 单线程为什么快？" → 纯内存操作 + 无锁竞争 + IO 多路复用
- "会话丢失会怎样？" → 用户历史上下文清空，体验降级但不报错——有兜底文案，属于可接受降级

---

## 8. 限流（滑动窗口）

**原理**：按用户 ID 记录请求时间戳列表，每次请求踢出窗口外旧记录，窗口内计数超阈值则拒绝。本项目配置：30 次 / 60 秒。

**选型权衡**：

| 算法 | 优点 | 缺点 |
| --- | --- | --- |
| 固定窗口 | 实现最简 | 窗口边界可瞬时 2 倍流量突刺 |
| 滑动窗口 | 平滑精确 | 存储时间戳，内存略高 |
| 令牌桶 | 允许突发、最平滑 | 实现与调参复杂 |

API 网关场景流量不大但要防刷，滑动窗口是精度与复杂度的平衡点。

**高频追问**：
- "为什么限流？" → LLM 调用有成本，防恶意刷接口烧钱；也防单用户挤占并发影响他人（配合 `MAX_CONCURRENT_PER_USER=3`）
- "分布式限流？" → 计数存 Redis 用 INCR+EXPIRE 或 Lua 脚本保证原子性

---

## 9. JWT + API Key 双认证

**原理**：
- **JWT**：服务端签发的带签名令牌，内含用户身份与过期时间，客户端每次请求带上，服务端验签免查库——解决"这次请求是谁"
- **API Key**：预分配的调用方标识，解决"哪个应用/渠道在调"

**为什么两个都要**：JWT 管会话级身份（会过期，泄露风险窗口短），API Key 管服务级准入（长期有效，便于按渠道统计和吊销）；前端再加访问密码门禁，三层纵深。密钥全部走环境变量，`.env` 在 `.gitignore`，不进仓库。

**高频追问**：
- "JWT 泄露了怎么办？" → 设短过期时间（本项目 120 分钟）；高安全场景加刷新令牌与黑名单
- "签名能伪造吗？" → 不能，HMAC/RSA 签名没有密钥算不出来，改一个字符验签就失败

---

## 10. Embedding 与 Reranker（双塔 vs 交叉编码）

**原理**：
- **Embedding（BGE-base-zh）**：双塔结构，query 和文档**分别**编码成向量，检索时算余弦相似度——快（文档向量可预先算好），但两者没有深度交互
- **Reranker（BGE-reranker）**：交叉编码器，把 query+文档**拼成一个序列**进模型直接输出相关性分——精度高，但每对都要跑一次模型，慢

**选型权衡**：这就是"召回用双塔、精排用交叉"的原因——双塔从全库粗筛 Top-N（快），交叉只对 Top-N 精排（准），精度与速度各取所长。中文场景选 BGE 是因为 C-MTEB 榜单中文效果领先且开源可本地部署。

**高频追问**：
- "为什么本地化部署？" → 首次远程加载 10-30s+ 且依赖外网；本地化后启动预热（先跑一次推理驻留内存），首问延迟 40s→8s
- "嵌入维度多少？影响什么？" → BGE-base 768 维；维度越高表达力越强但存储和计算越贵，base 版是效果与成本平衡点
