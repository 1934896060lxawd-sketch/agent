# 面试问题清单 — 汽车导购 AI 问答助手项目

> 面向岗位：AI Agent 开发工程师
> 用法：每题附「考察点」和「回答要点」，要点均与本项目真实实现对应，回答时结合具体数字和文件位置更有说服力。
> 建议：先自己口头作答，再对照要点查漏补缺。

---

## 一、项目整体与架构

**1. 用 1-2 分钟介绍一下你这个项目。**　⭐⭐⭐

- 考察点：表达能力和对项目的整体把控（必考开场题）
- 回答要点：一句话定位（ReAct Agent + RAG 的汽车导购问答系统）→ 解决什么问题（参数有出处、不编造、多轮不跑偏、流式无协议泄露）→ 技术栈（FastAPI + Streamlit + FAISS/BM25/BGE + GLM）→ 量化成果（首问延迟 40s→8s、25 组对抗用例、八轮回归验收）

**2. 为什么前端选 Streamlit，后端选 FastAPI，而不是前后端都用一个框架？**　⭐

- 考察点：技术选型的权衡能力
- 回答要点：职责分离——FastAPI 专注 API/认证/限流/SSE，便于独立压测和将来替换前端；Streamlit 快速搭建对话 UI，会话状态管理简单；两者通过 HTTP/SSE 解耦，前端可独立演进为 React/Vue

**3. 整体请求链路是怎样的？从用户输入到界面显示经过哪些环节？**　⭐

- 考察点：是否能完整讲清数据流
- 回答要点：浏览器 → Streamlit（密码门禁、会话管理）→ FastAPI（认证 → 限流 → 会话读写）→ ReAct Agent（组装 prompt → LLM → 拦截工具调用 → 执行工具 → 回填结果 → 再推理）→ 字符级清洗 → SSE 逐字推送 → 前端流式渲染

**4. 这个项目最大的技术难点是什么？你是怎么解决的？**　⭐⭐⭐

- 考察点：问题分析与解决能力（高频行为题）
- 回答要点：可从三个真实难点任选一个深入讲：① 工具调用 XML 泄露（架构上非流式拦截+字符级清洗）；② 短回复答非所问（上下文锚定）；③ 冷启动慢（模型本地化+启动预热）。讲清"现象 → 定位 → 方案 → 验证"四步

## 二、ReAct Agent 设计

**5. 什么是 ReAct 模式？你的 Agent 循环是怎么实现的？**　⭐⭐

- 考察点：Agent 基础范式理解
- 回答要点：ReAct = Reasoning + Acting，模型在"思考 → 选择工具 → 观察结果 → 再思考"中循环，直到给出最终回答；本项目在 `backend/agent/advisor.py` 中实现循环：每轮把工具结果回填 messages，直到模型不再发起工具调用或达到最大轮数

**6. 为什么首轮要强制 `tool_choice=required`？有什么副作用吗？**　⭐⭐

- 考察点：对 Function Calling 机制的深入理解
- 回答要点：目的是强制"先查后答"，杜绝模型凭训练记忆编造汽车参数（幻觉）；副作用是闲聊类消息（"你好"）也会触发工具调用，处理方式是工具层对无关查询返回友好的兜底结果，或在 prompt 中允许直接结束

**7. 你的 Agent 有哪几个工具？工具的颗粒度是怎么划分的？**　⭐

- 考察点：工具抽象设计能力
- 回答要点：5 个——知识库检索、车型对比、预算推荐、参数查询、购车成本计算；划分原则：一个工具对应一个明确的业务意图，入参简单（车型名/预算数值），返回结构化数据；避免"万能工具"导致模型选错

**8. 车名"四级模糊解析"是怎么做的？为什么需要它？**　⭐

- 考察点：容错设计
- 回答要点：用户可能输入"宋L""比亚迪宋L""song L"等不同写法；解析顺序：标准全称精确匹配 → 别名表 → 拼音 → 模糊匹配；保证工具调用时车型名能被可靠归一化，否则检索会频繁落空

**9. Agent 循环如果失控（一直调工具不停止）怎么办？**　⭐

- 考察点：健壮性设计
- 回答要点：设置最大循环轮数兜底；同一工具同参数重复调用时中断；超时控制；最终强制要求模型基于已有观察给出回答

**10. ReAct 和 Plan-and-Execute 有什么区别？你为什么选 ReAct？**　⭐

- 考察点：Agent 范式横向对比
- 回答要点：ReAct 逐步决策、灵活、适合工具少且路径短的场景；Plan-and-Execute 先规划全流程再执行，适合长链条任务但规划错误代价高；导购问答通常 1-3 次工具调用即可，ReAct 更简单可控、延迟更低

**11. 如果让你支持多 Agent 协作（比如一个查参数、一个算价格），你会怎么改？**　⭐

- 考察点：架构演进思维
- 回答要点：引入 Supervisor/Router 角色分发子任务；或保持单 Agent 但扩充工具集（当前架构已支持，工具注册式扩展）；讨论多 Agent 的上下文共享与成本问题

## 三、XML / 协议泄露问题（项目核心亮点）

**12. 什么是工具调用协议泄露？为什么会发生？**　⭐⭐⭐

- 考察点：对 LLM 输出机制的理解
- 回答要点：模型以文本形式输出 `<tool_call>{...}</tool_call>` 等标记来表达调用意图；如果直接把原始输出流式透传给前端，用户就会看到这些 XML 碎片，体验极差

**13. 你是怎么解决的？为什么不在前端过滤？**　⭐⭐⭐

- 考察点：架构分层意识（本项目的招牌题，务必吃透）
- 回答要点：在 Agent 层**非流式拦截**——工具调用阶段先完整收集模型输出、解析出工具调用并执行，确认不再含协议标记后，才把最终回答以**字符级流式**转发给 SSE；不在前端过滤是因为：① 标记可能跨 chunk 被截断，前端无法可靠识别；② 安全与体验逻辑应收敛在后端，前端保持"所见即所得"

**14. "跨 chunk 截断"具体指什么？流式场景下处理难点在哪？**　⭐

- 考察点：对流式传输细节的理解深度
- 回答要点：SSE 按 token 分片，`<tool_call>` 可能被拆成 `<tool_` + `call>` 两片，任何基于单片的正则都会漏；本项目通过"先完整收集再转发"从根本上规避了这个问题，代价是工具执行阶段无流式输出，用状态提示（"正在查询…"）补偿

**15. 怎么验证你的方案真的有效？**　⭐

- 考察点：质量意识
- 回答要点：构造 25 组对抗用例做回归——截断标记、嵌套标签、大小写变体、标记出现在正常文本中间等，断言最终输出零协议残留；测试脚本 `tmp_xmlcheck/stub_and_test_strip.py`

## 四、RAG 检索

**16. 你的 RAG 管线完整流程是怎样的？**　⭐⭐⭐

- 考察点：RAG 基础功
- 回答要点：文档切块（chunking）→ BGE 嵌入向量化 → FAISS 建索引；查询时：FAISS 向量召回 + BM25 关键词召回 → RRF 融合排序 → BGE Reranker 精排 → Top-K 注入 prompt

**17. 为什么要"向量 + BM25"混合检索？只用向量检索有什么问题？**　⭐⭐

- 考察点：对两种检索方式优劣的理解
- 回答要点：向量检索擅长语义相似（"适合家用的车"），但对精确型号名、参数数字（"宋L 662km"）不敏感；BM25 擅长精确词匹配；两者互补，RRF 融合避免单一召回方式的盲区

**18. RRF（倒数排名融合）的原理是什么？为什么不用加权分数融合？**　⭐⭐

- 考察点：算法理解
- 回答要点：RRF 对每条结果按其在各召回列表中的名次计算 `1/(k+rank)` 累加得分，不依赖原始分数的量纲；向量余弦相似度和 BM25 分数分布完全不同，直接加权难以调参，RRF 天然免疫量纲差异

**19. 已经有了融合排序，为什么还要加 Reranker 精排？**　⭐

- 考察点：对检索精度链路的理解
- 回答要点：向量/BM25 是"双塔"粗排，查询和文档分别编码，交互信息有限；Reranker 是交叉编码器，把查询和文档拼在一起算相关性，精度高但慢，所以只对粗排 Top-N 做精排，是精度与速度的权衡

**20. 文档切块（chunking）策略是什么？块太大/太小各有什么问题？**　⭐

- 考察点：RAG 工程细节
- 回答要点：按车型文档的自然结构切块并保留重叠；太大会稀释关键信息、超 token 上限；太小会切断上下文（如参数表与车型名分离）；可结合 `backend/rag/chunker.py` 说明具体参数

**21. 怎么评估你的检索效果好不好？**　⭐

- 考察点：评估意识（区分有没有做过真实 RAG 项目）
- 回答要点：人工构造问题集验证召回命中率；观察最终回答中参数是否与知识库一致；可扩展方向：召回率@K、MRR、端到端答案正确率

**22. RAG 和微调（Fine-tuning）有什么区别？为什么这个项目选 RAG？**　⭐⭐

- 考察点：LLM 基础知识（高频通用题）
- 回答要点：RAG 把知识放在外部库、可即时更新、有出处、成本低；微调把知识写进权重、更新贵且会遗忘；汽车参数/价格频繁变动，且要求"有出处不编造"，RAG 是正确选择

## 五、多轮对话与上下文

**23. 用户回复"需要""好的"，系统怎么知道他想干什么？**　⭐⭐⭐

- 考察点：上下文工程能力（本项目真实踩过的坑）
- 回答要点：上下文锚定机制——识别助手上一轮结尾的提议（如"要不要帮你算落地价？"），当用户短回复为肯定词时，把锚定的待办动作显式注入 prompt；同时过滤"谢谢""好的"等客套话避免污染历史

**24. 会话记忆是怎么存的？为什么用 Redis，不用内存字典？**　⭐

- 考察点：存储选型
- 回答要点：会话消息存 Redis，带 TTL 过期；Redis 支持多实例共享、重启不丢；本地开发无 Redis 时用 fakeredis 兜底，接口一致零改动

**25. 对话历史越来越长，token 超限怎么办？**　⭐

- 考察点：上下文窗口管理
- 回答要点：滑动窗口截取最近 N 轮；更早的内容可做摘要压缩；区分"系统事实"（用户预算、已锚定动作）与闲聊，优先保留事实

**26. 指代消解（"它的续航怎么样"中的"它"）怎么处理？**　⭐

- 考察点：多轮对话经典难题
- 回答要点：历史中保留最近讨论的车型实体，prompt 中提示模型先解析指代；车名模糊解析层兜底；八轮回归中有专项用例验证

## 六、后端工程

**27. SSE 和 WebSocket 有什么区别？为什么选 SSE？**　⭐⭐

- 考察点：网络协议基础
- 回答要点：SSE 是服务器到客户端的单向流，基于 HTTP、实现简单、天然支持断线重连；WebSocket 双向但复杂；LLM 流式输出只需单向推送，SSE 是最简方案

**28. 限流是怎么做的？滑动窗口和固定窗口/令牌桶有什么区别？**　⭐

- 考察点：高并发基础
- 回答要点：按用户维度滑动窗口计数（`RATE_LIMIT_REQUESTS / RATE_LIMIT_WINDOW_SECONDS`）；固定窗口有临界突刺问题；令牌桶平滑但实现复杂；滑动窗口精度与复杂度平衡

**29. 认证方案是什么？JWT 和 API Key 分别解决什么问题？**　⭐

- 考察点：安全设计
- 回答要点：JWT 用于会话级身份（有过期时间），API Key 用于服务级调用方标识；前端另有访问密码门禁层；密钥均走环境变量不入库

**30. 如果同时有 100 个用户提问，你的系统瓶颈在哪？怎么优化？**　⭐

- 考察点：性能分析思维
- 回答要点：瓶颈依次是：LLM API 并发与速率限制（外部依赖）→ Reranker 本地推理（CPU/GPU 算力）→ FAISS 检索（内存）；优化：请求排队与并发上限（已有 `MAX_CONCURRENT_PER_USER`）、异步 IO、Reranker 批处理/GPU 化、LLM 多 key 轮询

**31. 后端启动时做了哪些初始化？为什么首问延迟能从 40s 降到 8s？**　⭐⭐⭐

- 考察点：对冷启动优化的理解（项目量化亮点）
- 回答要点：嵌入模型和 Reranker 从 HuggingFace 远程加载改为本地 `models/` 目录加载；启动时预热（先跑一次推理把模型驻留内存）；FAISS 索引预加载；后端就绪约 2.5s

## 七、LLM 通用问题（高频基础题）

**32. 怎么减少大模型的幻觉？你项目里用了哪些手段？**　⭐⭐

- 回答要点：① 强制工具调用先查后答；② RAG 注入真实数据；③ prompt 中要求"参数必须来自检索结果"；④ 结构化工具返回而非自由生成

**33. Prompt 里 system / user / assistant 三种角色分别起什么作用？**　⭐

- 回答要点：system 定义行为边界与规则（工具使用约束、禁止编造）；user 是用户输入；assistant 历史用于维持多轮上下文与 few-shot 示例

**34. temperature 参数是干什么的？导购场景应该调高还是调低？**　⭐

- 回答要点：控制采样随机性；参数查询/推荐场景调低（接近 0）保证稳定准确；闲聊话术可略高

**35. Function Calling 的原理是什么？模型怎么"学会"调工具的？**　⭐⭐

- 回答要点：工具以 JSON Schema 描述随请求传给模型，模型训练时见过此类格式，输出特定结构的文本/标记表达调用意图；客户端解析后真正执行，结果回填再继续对话——模型本身不执行任何代码

**36. 流式输出是怎么实现的？token 是怎么一个个出来的？**　⭐

- 回答要点：LLM API 开启 stream 后按生成的 token 增量推送；后端逐 chunk 转发 SSE；前端边收边渲染；本项目特殊点：工具调用阶段不流式，最终回答阶段才流式

## 八、部署与运维

**37. 怎么让别人公网访问你本地跑的服务？Cloudflare 隧道的原理是什么？**　⭐

- 回答要点：cloudflared 在本机与 Cloudflare 边缘节点间建加密出站隧道，分配公网域名，无需公网 IP/端口映射；配合访问密码门禁控制准入

**38. 为什么要把模型本地化？直接在线加载有什么问题？**　⭐

- 回答要点：首次加载慢（10-30s+）、依赖外网稳定性、服务器迁移后重复下载；本地化后启动可控、可预热、离线可用

**39. 如果让你把这套系统部署到云服务器正式上线，你还需要补哪些东西？**　⭐

- 考察点：生产环境意识
- 回答要点：HTTPS/域名、进程守护（systemd/Docker）、日志与监控告警、真实 Redis、数据库持久化会话、LLM key 配额管理与降级策略、CI/CD

## 九、测试与质量

**40. 你怎么测试一个 AI 应用？传统单元测试够用吗？**　⭐⭐⭐

- 考察点：AI 测试方法论（新兴热门题）
- 回答要点：不够——模型输出非确定性，需要分层：① 确定性部分用传统断言（会话 CRUD 17 项断言、协议清洗 25 用例）；② 端到端用真实场景脚本跑多轮对话人工/半自动验收（八轮回归）；③ 对抗用例专测边界（泄露、越权、超长输入）

**41. 你的回归测试是怎么组织的？发现 bug 后怎么防止再犯？**　⭐

- 回答要点：每轮验收固化成可重复运行的脚本（`tmp_xmlcheck/` 下 e2e_live_test、test_session_crud、test_visitor_flow 等），修复后重跑全量；典型例子：密码框残留 bug 修复后纳入访客流程 8 项断言

## 十、开放与行为问题

**42. 这个项目如果继续做，你最想改进的三个点是什么？**　⭐

- 回答要点（展示思考深度）：① 引入评测集量化检索与回答质量；② 知识库自动更新管线（爬取新车参数）；③ 工具调用阶段也实现流式状态推送，进一步降低等待感

**43. 项目中有哪些是你独立完成的？遇到不会的东西怎么解决？**　⭐

- 回答要点：强调全流程独立（需求→架构→开发→测试→部署）；举例：XML 泄露问题从定位到方案设计的过程，体现排查能力（看日志 → 抓原始输出 → 分层隔离验证）

**44. 为什么想做 AI Agent 方向？**　⭐

- 回答要点：结合本项目——从"调用一次 LLM"到"让 LLM 自主使用工具完成任务"是质变，Agent 是 LLM 落地的主要形态；自己享受把模型能力工程化为可靠产品的过程

**45. 反问环节：你一般会问面试官什么问题？**　⭐

- 建议准备：团队当前 Agent 落地场景与技术栈 / 新人前 3 个月的期望产出 / 团队的模型评测与上线流程
