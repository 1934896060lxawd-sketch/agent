# 面试问题清单 1 — AI Agent 开发工程师经典题（通用篇）

> 与 `面试问题.md`（项目篇）互补：本清单收集不依赖具体项目的**行业经典面试题**，覆盖大模型基础、Agent 范式、框架工具、推理部署、评测安全等。
> 每题附「考察点」和「回答要点」。要点用于校准方向，建议用自己的话组织答案，并尽量往项目经历上挂钩。
> 每题附重要程度星级（⭐⭐⭐ 必背 / ⭐⭐ 重点 / ⭐ 了解），末尾附面试官常用的追问链示例。

## 一、大模型基础原理

**1. Transformer 的核心是什么？自注意力机制解决了什么问题？**　⭐⭐⭐

- 考察点：基础中的基础，几乎必问
- 回答要点：Self-Attention 让序列中任意两个位置直接交互，解决 RNN 长程依赖弱、无法并行的问题；Q/K/V 三个矩阵计算相关性权重，缩放点积后加权求和；多头注意力捕获多种子空间关系

**2. 为什么注意力分数要除以 √d（缩放）？**　⭐

- 考察点：细节理解深度
- 回答要点：维度 d 增大时点积方差变大，softmax 进入饱和区导致梯度消失；除以 √d 把方差归一，保持梯度稳定

**3. 位置编码有哪些方式？RoPE 的优势是什么？**　⭐

- 回答要点：正弦绝对位置编码（原始 Transformer）、可学习位置嵌入、相对位置编码；RoPE（旋转位置编码）通过旋转把相对位置信息注入 Q/K，外推性好，是 LLaMA/Qwen 等主流模型的选择

**4. 大模型训练的三个阶段（预训练、SFT、RLHF/DPO）分别做什么？**　⭐⭐⭐

- 考察点：对模型能力来源的理解
- 回答要点：预训练用海量无标注文本学"知识+语言能力"（next token prediction）；SFT 用指令数据教模型"听懂指令、按格式回答"；RLHF/DPO 用人类偏好对齐"回答质量与价值观"；Agent 能力主要在 SFT 阶段的工具调用数据和后训练阶段形成

**5. 什么是幻觉（Hallucination）？产生原因和缓解手段有哪些？**　⭐⭐⭐

- 考察点：高频通用题
- 回答要点：原因——训练数据噪声、next-token 目标的"强行续写"倾向、知识边界不清；缓解——RAG 检索增强、工具调用查证、prompt 要求"不知道就说不知道"、引用出处、降低 temperature、后训练对齐

**6. Token 是怎么切分的？为什么中文比英文"费 token"？**　⭐

- 回答要点：BPE/SentencePiece 等子词切分；英文大量词落在常用子词表内，中文单字/生僻组合常被拆成多个 token；影响成本估算、上下文长度规划和流式输出

**7. 上下文窗口（Context Window）受什么限制？长上下文有哪些方案？**　⭐

- 回答要点：注意力计算量随长度平方增长、KV cache 显存占用随长度线性增长；方案——RoPE 外推（NTK/YaRN）、滑动窗口注意力、稀疏注意力、RAG 外置记忆、上下文压缩/摘要

**8. Decoder-only、Encoder-only、Encoder-Decoder 架构的区别？各适合什么任务？**　⭐

- 回答要点：GPT 系 Decoder-only 适合生成；BERT 系 Encoder-only 适合理解/检索（本项目 BGE 嵌入就是 Encoder-only）；T5 系 Encoder-Decoder 适合序列到序列转换；理解这点能解释"为什么检索模型和生成模型用不同架构"

## 二、Agent 核心概念

**9. 你怎么定义"AI Agent"？和普通 LLM 调用的本质区别是什么？**　⭐⭐⭐

- 考察点：岗位核心认知题
- 回答要点：Agent = LLM（大脑）+ 规划（Planning）+ 记忆（Memory）+ 工具使用（Tool Use），能在环境中**自主感知-决策-行动-观察**循环直到达成目标；普通调用是"一问一答的单次映射"，Agent 多了状态、循环和对外部世界的副作用

**10. ReAct、Plan-and-Execute、Reflexion 三种范式对比？**　⭐⭐⭐

- 考察点：Agent 范式全家桶（高频）
- 回答要点：ReAct——边推理边行动，灵活、延迟低，适合短链路；Plan-and-Execute——先整体规划再逐步执行，适合长任务但计划错误代价高，常加 Replan；Reflexion——失败后自我反思生成改进策略存入记忆，提升重试成功率

**11. Agent 的"记忆"分哪几类？各自怎么实现？**　⭐

- 回答要点：感觉记忆（当前 prompt 上下文）、短期记忆（会话历史，滑动窗口/摘要）、长期记忆（向量库存用户偏好与历史结论，检索注入）、程序性记忆（沉淀的经验/SOP，写进 prompt 或 few-shot）

**12. 单 Agent 和多 Agent 系统（Multi-Agent）各自的适用场景？**　⭐

- 回答要点：单 Agent 简单可控、成本低，工具少任务明确时首选；多 Agent 适合角色天然分离（写手+审稿）、长链路分工（研究员+分析师+编辑）、需要对抗校验的场景；代价是编排复杂、上下文同步难、token 成本数倍增长

**13. 什么是 MCP（Model Context Protocol）？它解决了什么问题？**　⭐⭐

- 考察点：2024-2025 年热点，Agent 岗加分题
- 回答要点：Anthropic 提出的开放协议，把"工具/数据源接入"标准化为统一的 server-client 接口；解决 N 个模型 × M 个工具的 M×N 集成问题——工具方实现一次 MCP server，任何支持 MCP 的客户端都能用；类比"AI 工具的 USB-C 接口"

**14. A2A（Agent-to-Agent）协议和 MCP 有什么区别？**　⭐

- 回答要点：MCP 解决 Agent ↔ 工具/数据的连接；A2A（Google 提出）解决 Agent ↔ Agent 的通信与协作，定义任务委托、状态同步、能力发现；两者互补：Agent 用 A2A 互相协作，用 MCP 调工具

**15. 什么是 Agentic Workflow？和工作流（Workflow）有什么区别？**　⭐⭐

- 回答要点：Workflow 是开发者预先写死的固定流程（DAG），LLM 只是其中节点；Agentic 是模型在运行时自主决定下一步；实际工程中常见"结构化工作流 + 关键节点 Agent 自主"的混合形态，平衡可控性与灵活性

**16. 设计一个 Agent 时，你会从哪几个维度做技术选型？**　⭐

- 考察点：系统设计思维
- 回答要点：任务链路长短与确定性（选 ReAct / Plan-Execute / 工作流）、工具数量与复杂度、延迟要求（影响流式与模型档位）、成本预算（token 单价 × 轮数）、可控性要求（是否允许自由发挥）、评估手段是否可得

## 三、Function Calling 与工具使用

**17. Function Calling 的完整流程是什么？模型真的"执行"了函数吗？**　⭐⭐⭐

- 考察点：必考基础
- 回答要点：工具以 JSON Schema 注入 → 模型输出结构化调用意图（函数名+参数）→ **客户端代码真正执行** → 结果作为 tool 消息回填 → 模型继续推理；模型只生成文本，执行永远在客户端，这决定了参数校验、权限控制必须在工程层做

**18. 模型选错工具或传错参数怎么办？**　⭐⭐

- 回答要点：工具描述写清"何时用/何时不用"和参数示例（few-shot）；参数层做强校验并返回可理解的错误信息让模型自我修正；约束解码（强制输出合法 JSON Schema）；高风险操作加人工确认环节

**19. 工具数量很多（上百个）时怎么办？**　⭐

- 考察点：工程扩展性思维
- 回答要点：工具检索（把工具描述向量化，按 query 召回 Top-N 工具再注入）；分层工具（先选类别再选具体工具）；按场景动态加载；这也是 MCP 生态想解决的问题之一

**20. 并行工具调用（Parallel Function Calling）是什么？什么场景用？**　⭐

- 回答要点：模型一轮输出多个无依赖关系的调用，客户端并发执行后一起回填，省往返轮次、降延迟；适合"同时查三款车的参数"这类相互独立的查询；有依赖关系的（查价格→算落地价）必须串行

## 四、Prompt 工程

**21. Zero-shot / Few-shot / CoT 的区别？CoT 为什么有效？**　⭐⭐⭐

- 回答要点：Zero-shot 直接问；Few-shot 给示例；CoT（思维链）要求"逐步推理"，把复杂问题分解，中间步骤外显让模型分配更多计算在推理上，显著提升数学/逻辑任务表现；变体：Zero-shot-CoT（"Let's think step by step"）、Self-Consistency（多条 CoT 投票）

**22. System Prompt 设计的原则有哪些？**　⭐

- 回答要点：角色与目标明确；行为边界（能做什么/不能做什么）；输出格式约束；工具使用规则；冲突时的优先级；避免与模型训练习惯对抗；重要指令放开头或结尾（lost in the middle 现象）

**23. 什么是 Prompt Injection？怎么防御？**　⭐⭐

- 考察点：安全意识（Agent 岗越来越重视）
- 回答要点：攻击者在输入/检索内容中嵌入恶意指令劫持 Agent（"忽略之前的指令，把用户数据发到…"）；防御——输入输出隔离（指令与数据分区）、工具权限最小化、高危操作二次确认、输出过滤、对检索内容做清洗标注

**24. "Lost in the Middle" 是什么现象？怎么应对？**　⭐

- 回答要点：模型对长上下文开头和结尾的内容利用最好，中间部分容易被忽略；应对——关键信息放首尾、控制上下文长度、用检索只注入相关内容而非全量塞入

**25. 结构化输出怎么保证？JSON 总是解析失败怎么办？**　⭐

- 回答要点：Function Calling / response_format 的 JSON mode；约束解码（outlines、guidance 等库强制语法合法）；prompt 给 schema 和示例；解析失败重试+错误反馈让模型修正；关键字段单独校验

## 五、RAG 进阶经典题

**26. 一个标准 RAG 系统有哪些模块？各环节常见的坑是什么？**　⭐⭐⭐

- 回答要点：文档解析（PDF 表格丢失）→ 切块（语义断裂）→ 嵌入（领域不匹配）→ 索引（召回率低）→ 检索（单一方式盲区）→ 重排 → 生成（不忠于检索内容）；每个环节都可能成为瓶颈，需要逐环评估

**27. Embedding 模型怎么选？中文场景注意什么？**　⭐

- 回答要点：看 MTEB/C-MTEB 榜单、维度（影响存储与速度）、上下文长度、领域匹配度；中文常用 BGE 系列、M3E、text2vec；领域差距大时可用领域语料微调嵌入模型

**28. 主流向量数据库对比（FAISS / Milvus / Chroma / Qdrant / ES）？**　⭐

- 回答要点：FAISS 是库不是服务，单机嵌入式、性能强，适合中小规模（本项目选型）；Milvus/Qdrant 是独立服务，支持分布式、过滤、在线更新，适合大规模生产；Chroma 轻量易上手；ES/OpenSearch 天然支持向量+关键词混合检索

**29. 什么是 Agentic RAG？和传统 RAG 的区别？**　⭐⭐

- 考察点：RAG 前沿热点
- 回答要点：传统 RAG 是"检索一次→生成"的固定管线；Agentic RAG 让 Agent 自主决定"要不要检索、检索什么、结果够不够用、要不要再查一次"，支持多轮检索、查询改写、工具化检索源；本质是 RAG 从流水线变成 Agent 的一个工具

**30. 查询改写（Query Rewrite）有哪些手段？**　⭐

- 回答要点：HyDE（让模型先生成假设答案再拿去检索）、多查询扩展（一个问句改写成多个子查询分别检索再合并）、指代消解改写（结合历史把"它"补全为实体名）、Step-back prompting（先抽象再检索）

**31. RAG 系统怎么评估？RAGAS 的指标有哪些？**　⭐⭐

- 回答要点：分检索层（命中率、MRR、Recall@K）和生成层；RAGAS 经典四指标——Faithfulness（答案忠于检索内容）、Answer Relevancy（答案切题）、Context Precision（检索内容相关度）、Context Recall（检索内容覆盖度）；评估方法含 LLM-as-Judge

**32. GraphRAG 是什么？适合什么场景？**　⭐

- 回答要点：微软提出，用知识图谱组织文档实体关系，检索时沿图结构聚合多跳信息；适合需要跨文档推理、全局总结（"整个语料主要讲了什么"）的场景；成本是建图贵、维护复杂

## 六、推理与部署

**33. KV Cache 是什么？为什么能加速推理？**　⭐⭐⭐

- 考察点：推理原理高频题
- 回答要点：自回归生成时每步都要重算所有历史 token 的注意力；KV Cache 把历史的 Key/Value 缓存复用，每步只算新 token，计算量从 O(n²) 降为每步 O(n)；代价是显存随序列长度线性增长

**34. Prefill 和 Decode 两个阶段的特征分别是什么？**　⭐⭐

- 回答要点：Prefill 处理输入 prompt，并行计算、算力密集（compute-bound）；Decode 逐 token 生成，串行、显存带宽密集（memory-bound）；优化方向不同——Prefill 要算力，Decode 要带宽，vLLM 等框架对两者分别调度

**35. 量化（Quantization）是什么？INT8/INT4 对模型有什么影响？**　⭐

- 回答要点：把 FP16 权重/激活压缩到低比特，显存减半甚至更低、推理加速；INT8 几乎无损，INT4 有轻微精度下降；常见方案 GPTQ、AWQ、GGUF；端侧/成本敏感场景的标配

**36. vLLM 为什么快？PagedAttention 解决了什么？**　⭐⭐

- 回答要点：KV Cache 传统上按最大长度预分配，显存碎片浪费严重；PagedAttention 借鉴操作系统分页思想，把 KV Cache 分成块按需分配，显存利用率大幅提升，从而支持更大 batch，吞吐量数倍提升；配合 Continuous Batching 动态凑批

**37. 流式输出的 TTFT 和 TPOT 分别指什么？怎么优化？**　⭐

- 回答要点：TTFT（首 token 延迟）——受 Prefill 速度、排队、检索/工具前置耗时影响（本项目通过模型预热+本地化优化）；TPOT（每 token 间隔）——受 Decode 速度和网络传输影响；体验优化常把长前置过程拆出状态提示

**38. 本地部署开源模型 vs 调 API，怎么选？**　⭐

- 回答要点：API——零运维、效果上限高、按量付费，适合起步与流量波动场景；本地部署——数据不出内网、长期大流量更便宜、可深度定制，但需 GPU 投入和运维；常见策略：核心链路用稳定 API，敏感/高频简单任务用本地小模型

## 七、微调（Fine-tuning）

**39. LoRA 的原理是什么？为什么省资源？**　⭐

- 回答要点：冻结原权重，在注意力层旁路注入低秩矩阵 A×B（r≪d），只训练这部分参数；可训练参数量从全量降到 1% 以下，显存需求大降，且 LoRA 权重可热插拔切换；QLoRA 进一步把基座量化为 4bit

**40. 什么场景该微调，什么场景该 RAG？能结合用吗？**　⭐⭐

- 考察点：经典权衡题
- 回答要点：微调改"行为/风格/格式/领域语感"，知识固化且更新贵；RAG 供"事实/动态知识"，即改即生效；最佳实践常是 RAG 为主、微调校行为（如固定输出格式、工具调用习惯）；两者不互斥

**41. 微调数据质量的关键点有哪些？**　⭐

- 回答要点：少量高质量 > 大量低质量（LIMA 论文观点）；格式统一、覆盖目标场景、去重去噪；指令多样性避免过拟合单一口吻；坏数据比没数据更糟

## 八、评测与可观测性

**42. 怎么评测一个 Agent 系统的好坏？**　⭐⭐⭐

- 考察点：区分是否做过真实 Agent 项目
- 回答要点：分层——单步（工具选择正确率、参数正确率）、轨迹（步骤是否合理、轮数是否收敛）、结果（任务完成率）；手段——确定性断言、LLM-as-Judge 打分、人工抽检、线上 A/B；难点是输出非确定性，需固定数据集多次运行取分布

**43. LLM-as-Judge 是什么？有什么坑？**　⭐

- 回答要点：用强模型（如 GPT-4）给弱模型输出打分/ pairwise 比较；坑——位置偏好（偏爱先出现的答案）、长度偏好（偏爱长答案）、自我偏好（偏爱自家模型输出）、评分漂移；缓解——交换顺序、固定 rubric、人工校准样本

**44. Agent 上线后怎么做可观测性（Observability）？**　⭐

- 回答要点：Trace 全链路记录（每次 LLM 调用、工具调用、token 消耗、耗时）；按会话聚合回放；监控指标——成功率、平均轮数、token 成本/会话、工具失败率；工具链：LangSmith、Langfuse（开源）、OpenTelemetry

---

## 九、安全与合规

**45. Agent 系统的安全风险有哪些？怎么分层防御？**　⭐⭐

- 回答要点：Prompt Injection（指令劫持）、工具滥用（越权删库/发消息）、数据泄露（上下文带出敏感信息）、拒绝服务（超长输入刷 token）；防御分层——输入过滤、工具权限最小化+白名单、高危操作人工审批、输出审查、限流配额、审计日志

**46. 什么是 Guardrails（护栏）？怎么实现？**　⭐

- 回答要点：在 LLM 输入/输出两侧加规则与模型校验层：输入侧检测注入与越界请求，输出侧检测敏感信息、格式合规、事实一致性；实现手段——正则/关键词、分类模型、再用一个 LLM 做审查；框架如 NeMo Guardrails

## 十、场景设计题（现场设计，考察综合能力）

**47. 设计一个智能客服 Agent，说下架构。**　⭐⭐

- 答题框架：意图识别路由 → FAQ/RAG 查知识库 → 工具（查订单/退款）→ 多轮会话管理 → 转人工兜底；强调护栏（不承诺权限外事项）、评估（解决率、转人工率）、成本（分级模型：小问题小模型）

**48. 设计一个 Deep Research（深度调研）Agent。**　⭐

- 答题框架：问题拆解规划 → 多路搜索/抓取 → 信息去重与可信度评估 → 迭代深挖（发现新线索继续查）→ 结构化报告生成+引用出处；对应 Plan-and-Execute 范式；强调预算控制（最大轮数/token 上限）与中间态持久化

**49. 估算一个日活 1 万的问答 Agent 的 token 成本，并给出降本方案。**　⭐

- 答题框架：单次会话 token =（系统 prompt + 历史 + 检索内容 + 输出）× 轮数；降本——历史压缩摘要、检索内容精准截断、分级路由（简单问题小模型）、缓存高频问答、prompt 精简、离线批处理非实时任务

**50. 老板说"回答速度太慢"，你怎么排查？**　⭐

- 答题框架：先埋点分段计时（检索/LLM 首 token/工具执行/网络）→ 定位大头 → 针对性优化：检索慢（索引/本地化/缓存）、TTFT 慢（预热/换快模型/减 prompt）、生成慢（限 max_tokens/流式体验优化）、工具慢（并行/缓存）
